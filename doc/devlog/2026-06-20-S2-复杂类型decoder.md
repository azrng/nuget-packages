# Azrng.NMaxCompute S2：复杂类型 decoder（TIMESTAMP/JSON/ARRAY/MAP/STRUCT）

日期：2026-06-20
任务 ID：T018
关联：S2 阶段（plan: `azrng-nmaxcompute-python-python-c-pure-catmull.md`）；承接 T017（S1）

## 背景

S1 已打通 Tunnel wire 解码（基础类型 + 单记录 CRC32C 校验）。
S2 目标：补齐 TIMESTAMP / TIMESTAMP_NTZ / JSON，以及 ARRAY / MAP / STRUCT 递归 decoder，
加上复合类型字符串解析，使类型系统基本完整，为后续接入 Tunnel 取数链路铺平道路。

## 完成范围

### S2-1 TIMESTAMP / TIMESTAMP_NTZ / JSON decoder

- `Tunnel/Types/TimestampJsonDecoder.cs`
  - `TimestampDecoder`：秒（sint64）+ 纳秒（sint32），两部分分别计入 CRC。
    对应 PyODPS `_read_field` 中 `types.timestamp` 分支：`l_val * 1000` 传入 `from_milliseconds`，
    等价于 l_val 是 Unix 秒；nano_secs 为额外纳秒。
    - `LocalInstance`（默认）：`ToLocalTime()`，对齐 PyODPS `local_timezone=True`
    - `UtcInstance`（TIMESTAMP_NTZ）：保持 UTC
    - 纳秒按 100ns 取整（.NET tick 精度限制），越界秒夹到 DateTimeOffset 边界
  - `JsonDecoder`：length-delimited UTF-8，返回原始 JSON 文本字符串（ADO.NET `GetString` 友好）

### S2-2 ARRAY / MAP / STRUCT 递归 decoder

- `Tunnel/Types/CompositeDecoders.cs`
  - `ArrayDecoder(ITypeDecoder)`：读 uint32 长度，逐元素 [null marker, 值]
  - `MapDecoder(keyDecoder, valueDecoder)`：读 keys 数组 + values 数组，zip 成字典
  - `StructDecoder(names[], decoders[])`：逐字段 [null marker, 值]，按声明顺序
  - **关键 CRC 语义**：size 和 null marker **不**计入 CRC，只有叶子值（由元素 decoder 内部 update）计入。
    严格对齐 PyODPS `_read_array` / `_read_struct`——若把 size/marker 也计入，record 级 CRC 必然校验失败。

### S2-3 类型字符串解析 + Factory 接入

- `Tunnel/Types/TypeStringParser.cs`：对应 PyODPS `parse_composite_types`
  - 词法分析：`<` / `(` 开括号、`>` / `)` 闭括号、`,` 分隔、`` ` `` 引用字段名
  - 栈折叠：`array<T>` / `map<K,V>` / `struct<n1:T1,n2:T2>`，支持任意嵌套
  - struct 字段名解析：`name:type`，支持反引号包裹
- `Tunnel/Types/TypeDecoderFactory.cs` 重构：
  - 拆出 `GetPrimitiveDecoder`（仅查表）与 `GetDecoder`（含复合解析路由）
  - `decimal(p,s)` 短路（避免被当作复合类型）
  - 新增 `timestamp` / `timestamp_ntz` / `json` 路由

### S2-4 DataReader 类型映射扩展

- `MaxComputeDataReader.ResolveType`：
  - `timestamp` / `timestamp_ntz` → `DateTimeOffset`
  - `json` → `string`
  - `array<...>` → `object[]`，`map<...>` → `IDictionary`，`struct<...>` → `object[]`

## 关键设计决策

1. **CRC 语义严格对齐 PyODPS**：复合类型内部的 size / null marker 不计入 CRC。
   这是 S2 最易踩坑的点，用 `TunnelRecordCompositeTest.Record_WithArrayColumn_CorruptLeaf_ThrowsCrc`
   反向验证（故意让叶子值与 CRC 不符，确认会抛错）。
2. **JSON 返回原始字符串**而非解析对象：ADO.NET `GetString` 直接可用，避免 `JsonElement` 互操作问题；
   调用方需要结构化对象时自行 `JsonDocument.Parse`。与 PyODPS 的 `json.loads` 在「值」上等价（都是文本），
   仅 CLR 表示形式不同。
3. **TIMESTAMP 秒语义**：依据 PyODPS `to_datetime(l_val * 1000)`（`from_milliseconds`）反推，
   l_val 是 Unix 秒而非毫秒。无法用真实集群验证，已在代码注释和 README 风险说明标注。
4. **NULL 处理分两层**：record 级 NULL = 缺失 field index（S1 已实现）；复合类型内部 NULL = 前置 marker。
   无独立 bitmap。

## 已知缺口（留给后续阶段）

- TIMESTAMP 的秒/纳秒单位未用真实集群验证（依赖 S1 收尾的 Tunnel 接入后联调）
- 复合类型 decoder 已就绪但**未接入运行时**（同 S1，等 `OpenDataReaderAsync`）
- TIMESTAMP 纳秒精度受限于 .NET tick（100ns），与 PyODPS 纳秒精度有差
