# 文档规则

## 适用范围

- 作用域：设计文档、契约说明、设计系统与开发文档维护
- 触发场景：涉及阶段 0 设计、契约说明、`doc/requirement.md` 时阅读

### 阅读摘要
- 建议阅读：整理需求、补设计文档、维护契约、更新设计系统
- 可先跳过：纯界面实现、纯服务端逻辑、纯部署配置调整
- 优先查看：阶段 0 设计文档、契约说明规范、文档更新要求

### 常见任务入口
- 新增功能方案：先看阶段 0 设计文档
- 更新字段、接口、DTO 或 schema：先看契约说明规范
- 更新设计系统或通用说明：先看设计系统文件规范与文档目录约定

---

## 主动建议规则
- 发现需求描述缺少角色、流程、字段、权限、边界场景、验收标准或非功能要求时，应主动补充为待确认项
- 发现实现方案与设计文档、接口契约、需求基准或分层架构不一致时，应主动提醒并建议同步修订
- 维护文档时优先更新已有事实来源，不新增平行说明；确需新增文件时，应说明新增原因和归档位置
- 对不确定的业务事实、接口字段和运行方式，只能标记为待确认或引用已验证来源，禁止写成确定结论

---

## 阶段 0 设计文档

### 阶段 0 — 设计文档（新功能强制前置）

**触发条件**：用户提出新功能需求

**执行方式**：由用户与 AI 对话协作产出，用户最终确认定稿

**产物**：`doc/design/设计文档.md`（全项目一份，新功能以新章节追加，不新建文件）

**文档必须包含以下四个部分**：

| 部分 | 内容要求 |
| ---- | -------- |
| 架构设计 | 模块划分、层职责、依赖方向、涉及的技术决策说明 |
| 功能需求 | 接口列表、核心业务规则、权限要求、关键状态说明 |
| 接口契约 | 请求参数、响应结构、错误码、示例数据 |
| 处理流程 | 核心处理链路、层间协作方式、边界场景、异常处理方式 |

**门控规则**：
- 用户明确确认设计文档后，才允许进入阶段 1。
- AI 在此阶段只输出文档内容，不写任何实现代码。

## 接口契约规范

### 契约要求

- **唯一来源**：IApplication 层 DTO、宿主层公开接口定义与相关请求 / 响应模型。
- **验证方式**：通过 Swagger UI 展示和验证契约，不把 Swagger 文档本身当作手工维护源。
- **变更规则**：接口契约变更时，必须先修改 DTO 或接口定义，再同步修改 Application、Domain 映射、测试与相关文档。
- **回退规则**：后续阶段发现契约缺口时，按根 `AGENTS.md` 的纠错与回退机制回到契约变更流程，不在实现层绕过契约。
- **文件结构建议**：按业务域拆分 DTO 和接口定义，避免将全部契约堆叠在单个文件中。

## 文档更新要求
以下内容发生变化时，必须同步更新相关文档：
- 命令、脚本或启动方式
- 环境变量
- 配置项
- 构建、发布或部署流程
- 数据库结构、迁移或初始化方式
- 对外使用方式或接口约定

优先更新已有文档，不新增重复说明。

### 文档目录约定
- `doc/requirement.md`：需求文档。若文件已存在，AI 应基于用户新增需求进行补充或修订；若文件不存在，AI 必须根据用户输入先自动生成详细初稿。该文件仅承载业务需求，不写入设计说明或开发过程记录
- `doc/design/设计文档.md`：项目设计文档，全项目一份。包含架构设计、功能需求、接口契约与处理流程。由用户与 AI 协作产出，用户确认后作为开发的唯一需求基准。AI 可在文件中追加新功能章节，不删除已有内容，不新建平行文件

### requirement 要求
- 若 `doc/requirement.md` 为模板初始状态，AI 应根据用户输入将占位内容替换为项目真实需求，而不是长期保留“待补充”空壳。
- 需求文档建议固定包含以下章节：项目背景、目标与范围、用户角色、接口或模块清单、核心业务流程、关键数据与字段、权限与可见范围、边界场景与异常、非功能要求、待确认项。
- 自动生成需求文档时，优先写清业务事实、流程与边界，不提前写入技术实现方案。

### 回顾分析记录格式
- 适用场景：同类问题因规则缺口在同一任务中连续两次阻塞交付，或跨任务重复出现后，触发简短回顾分析时使用
- 内容保持简短，建议至少包含：
  - 错误现象
  - 触发场景
  - 根因
  - 本次补充或修改的规则
  - 后续如何避免或验证不再复发
